# Code Apps: Architecture

---

## For consultants

A **Code App** is a fully custom React single-page application (SPA) that runs inside the Power Platform. It gives you complete control over the UI and code while the platform handles identity, hosting, and data connectivity for you.

**How it compares to the other options:**

| | Code App | Canvas App | PCF Component |
|---|---|---|---|
| What you build | Full React SPA | Low-code screens with formulas | A single custom UI component |
| Code control | Complete | None (no custom code) | Complete, but scoped to one field/column |
| Where it lives | Standalone app in a Solution | Standalone app in a Solution | Embedded inside a Canvas or Model-driven App |
| Best for | Complex custom apps | Rapid business apps, forms | Replacing a single field with richer UI |

**The short version:** use a Code App when you need a real web app вҖ” routing, complex state, custom data visualisations, or full design freedom вҖ” but want the platform to take care of authentication and hosting.

**What the platform gives you for free:**
- Entra ID (Azure AD) authentication вҖ” users sign in with their Microsoft 365 account, no code required from you
- Secure hosting via the Power Apps CDN
- Automatic connection to the Power Platform connectors your app declares

**What you are responsible for:**
- All React code, components, and state management
- Defining which data sources (connectors) the app needs
- Following security rules (see `security.md`)

**The two files you must never edit manually:**
- `power.config.json` вҖ” the platform's own manifest; editing it by hand breaks the app
- Everything under `src/Services/` and `src/Models/` вҖ” auto-generated from your data source declarations; regenerate them with `pac code` commands instead

**Development flow in one sentence:** write React code locally вҶ’ push to the platform with `pac code push` вҶ’ test inside the real Power Apps player with real authentication.

## Technical specification

### Architectural components

A Code App is built from four interacting parts.

#### 1. `power.config.json`

The platform manifest file at the root of the project. It records:
- The app's unique identifier and display name
- The list of declared data sources (connectors) and their connection references
- Build output configuration

**Do not edit this file manually.** All modifications must go through `pac code` CLI commands. Manual edits can corrupt the app's registration in the Power Platform environment and break deployment.

Reference: [Power Apps Code Apps overview](https://learn.microsoft.com/en-us/power-apps/developer/code-apps/overview)

#### 2. `@microsoft/power-apps` SDK

An npm package that provides the bridge between your React code and the Power Apps host. It exposes:
- `PowerAppsHost` вҖ” the top-level context object injected at runtime
- Typed connector service proxies (generated into `src/Services/`)
- Type definitions for connection references and environment context

Install it via npm as part of `pac code create` scaffolding. Do not replace or mock this SDK in production code.

#### 3. PAC CLI (`pac`)

The Power Platform CLI is the toolchain for all Code App operations:

```bash
# Scaffold a new Code App
pac code create --name MyApp

# Add a data source / connector
pac code add-data-source --environment <env-id> --connector <connector-id>

# Push local changes to the platform
pac code push

# Authenticate to an environment
pac auth create --environment <environment-url>
```

The CLI regenerates `src/Services/` and `src/Models/` automatically whenever data sources change. This is why you never edit those folders by hand.

**Deprecation notice:** `pac code` sub-commands are being superseded by the npm-based CLI (available from `@microsoft/power-apps` v1.0.4+). Prefer the npm CLI for new projects.

Reference: [PAC CLI code reference](https://learn.microsoft.com/power-platform/developer/cli/reference/code)

#### 4. Power Apps Host

The runtime container provided by the platform. When a user opens the app in a browser or the Power Apps mobile client, the host:
1. Authenticates the user via Entra ID (OAuth 2.0 / OIDC) вҖ” no auth code required in your app
2. Loads your compiled React bundle from the CDN
3. Injects the `PowerAppsHost` context object into your app's entry point
4. Resolves connection references to live connector endpoints
5. Proxies all connector calls through the platform's secure gateway, enforcing DLP policies

Your code never receives raw credentials or bearer tokens. All data access is mediated by the host.

Reference: [Code Apps architecture](https://learn.microsoft.com/power-apps/developer/code-apps/architecture)

### Project folder structure

```
my-app/
в”ңв”Җв”Җ power.config.json          # Platform manifest вҖ” DO NOT edit manually
в”ңв”Җв”Җ package.json
в”ңв”Җв”Җ tsconfig.json
в”ңв”Җв”Җ src/
в”Ӯ   в”ңв”Җв”Җ index.tsx              # Entry point вҖ” mounts React app
в”Ӯ   в”ңв”Җв”Җ App.tsx                # Root component
в”Ӯ   в”ңв”Җв”Җ Services/              # AUTO-GENERATED вҖ” DO NOT edit manually
в”Ӯ   в”Ӯ   в””в”Җв”Җ DataverseService.ts
в”Ӯ   в”ңв”Җв”Җ Models/                # AUTO-GENERATED вҖ” DO NOT edit manually
в”Ӯ   в”Ӯ   в””в”Җв”Җ account.ts
в”Ӯ   в”ңв”Җв”Җ components/            # Your components
в”Ӯ   в”ңв”Җв”Җ features/              # Feature-scoped modules
в”Ӯ   в””в”Җв”Җ hooks/                 # Custom React hooks
в””в”Җв”Җ public/
```

`src/Services/` contains typed async service classes that wrap each declared connector. `src/Models/` contains TypeScript interfaces generated from the connector's schema. Both are regenerated every time you run `pac code add-data-source` or the equivalent npm CLI command.

### Development workflow

```
1. pac auth create --environment <url>
        в”Ӯ
        в–ј
2. pac code create --name MyApp
   (scaffolds project, installs SDK, creates power.config.json)
        в”Ӯ
        в–ј
3. npm run start
   (local dev server вҖ” hot reload, no real auth)
        в”Ӯ
        в–ј
4. pac code add-data-source ...
   (regenerates src/Services/ and src/Models/)
        в”Ӯ
        в–ј
5. Develop features locally
        в”Ӯ
        в–ј
6. npm run build
   (production build вҖ” must pass before push)
        в”Ӯ
        в–ј
7. pac code push
   (deploys compiled bundle to Power Platform environment)
        в”Ӯ
        в–ј
8. Test in real Power Apps player
   (real Entra ID auth, real connectors)
```

**Important:** always test with `pac code push` + real player before delivery. Local dev server uses mock context; it does not replicate real authentication or DLP enforcement.
